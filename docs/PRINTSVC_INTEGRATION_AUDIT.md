# Native Print-Service Integration Audit

This audit tracks Deneb code that used to compensate for the stock Python
`marlindriver` shape. The migration rule is that every remaining boundary must
either be a thin client of native `deneb-printsvc`, shared Deneb print-control
code, or a behavior owned directly by native `deneb-printsvc`. Compatibility
with the old Python driver is not a reason to keep duplicated behavior.

| Integration | Placement decision | Native owner | Compatibility boundary | Removal condition | Evidence | Remaining proof |
| --- | --- | --- | --- | --- | --- | --- |
| LCD `backend_comm` | Client via shared helpers. | Shared `common/print` helpers plus native `deneb-printsvc` status and command ZMQ endpoints. | `ui/src/backend_comm.c` remains the touchscreen transport and display adapter. It must use shared helpers for route selection, status state, pending-job dispatch, pending-job file names, command formatting, print file naming, abort-display context, lifecycle transition classification, preheat event classification, and state rules. | Keep only the transport/display adapter. Remove local lifecycle or pending-job reinterpretation when each status/action decision has a shared helper or service-owned field. | `backend_comm.c` includes `print_backend_route.h`, `status_state.h`, `pending_job_dispatch.h`, `pending_job_file.h`, `command_format.h`, `print_job_file.h`, and `print_state_rules.h`; LCD abort-display context calls `deneb_status_state_has_abort_context`, and lifecycle/preheat logging uses `deneb_status_state_transition_from_pair` plus `deneb_status_state_preheat_events`. | Hardware LCD interaction still needs proof for queued Cura jobs, start, pause/resume, abort, completion, and stale-state recovery. |
| web `backend_zmq` | Client via shared helpers. | Shared `common/print` helpers plus native `deneb-printsvc` status and command ZMQ endpoints. | `web/src/backend_zmq.c` remains the web/API transport cache and status adapter. | Keep transport cache only. Remove duplicate status naming, print-context retention, lifecycle transition classification, preheat event classification, and pending-job interpretation when shared helpers cover the field. | `backend_zmq.c` includes `print_backend_route.h`, `status_state.h`, `pending_job_dispatch.h`, `pending_job_file.h`, `print_history.h`, `print_job_file.h`, `command_format.h`, and `print_state_rules.h`; lifecycle/preheat logging uses `deneb_status_state_transition_from_pair` plus `deneb_status_state_preheat_events`, print history consumes the shared transition completion label, and cached status JSON uses the same request/native-aware status label as the REST/cluster adapters. | Web UI user-flow proof is still required on hardware, especially stale pending jobs and abort/completion transitions. |
| REST `api_print_job` | Client via shared helpers. | Shared print action, pending-job registration, job summary, and state-rule helpers. Native `deneb-printsvc` owns actual print execution. | The REST endpoint accepts stock-shaped UM API v1 actions and uploads, then delegates planning/dispatch through shared helpers. | Remove endpoint-specific action branching once all actions are expressed through `print_action_dispatch` and service-owned status fields. | `api_print_job.c` includes `pending_job_registration.h`, `print_action_dispatch.h`, `print_job_summary.h`, `pending_job_file.h`, `print_job_file.h`, and `print_state_rules.h`. | Native hardware proof still needs upload/start/abort/completion through web/API without stale pending state. |
| Cura `api_cluster` | Client via shared helpers. | Shared pending-job, print profile, job summary, and state-rule helpers. Native `deneb-printsvc` owns execution through `JOB`. | Cura LAN compatibility remains as an API adapter for Ultimaker cluster routes and conflict/pending-job semantics. | Keep Cura protocol surface, but remove duplicate status and queue interpretation after all conflict/start decisions use shared helpers and native service fields. | `api_cluster.c` includes `pending_job_file.h`, `print_job_summary.h`, `print_profile.h`, `command_format.h`, and `print_state_rules.h`; cluster action routing now uses `deneb_print_cluster_action_plan` so pending-job continue/cancel versus normal pause/resume/abort routing is owned by shared native rules. 2026-06-14 Cura 5.13 target proof on package `ff49e86b` covered discovery label, `.ufp` extraction before registration, material-mismatch prompt, Cancel, Continue/start, completion, pause/resume, and cancel/abort back to idle. | Cura stale-state recovery, progress/time reporting, and broader failure modes remain open. |
| REST `api_printer` | Client via shared helpers. | Shared motion, G-code command formatting, printer status response helpers, and shared heat-target state rules. | The endpoint still exposes stock-shaped printer controls while formatting safe native commands through helpers. | Keep route compatibility only. Move any raw G-code construction into shared helpers or native service policy before promotion. | `api_printer.c` includes `gcode_command.h`, `manual_motion.h`, and `printer_status_response.h`; `printer_status_response.c` uses `print_state_rules.h` so UM printer root, `/printer/bed`, and `/printer/bed/pre_heat` report bed preheat activity from the same heat-target rule instead of hard-coded inactive JSON. | Physical motion and heat endpoints need supervised hardware proof with homing and safety plans. |
| conflict and preheat bridges | Client via shared helpers. | Shared pending-job dispatch, print action dispatch, print state rules, and native service active/stop fields. | LCD conflict screens, REST state actions, and Cura pending-job routes remain separate UI/API adapters. | Remove bridge-specific lifecycle logic when native service publishes enough active/preheat/stop state for every client to render the same decision. | LCD/web adapters use `pending_job_dispatch.h`, `pending_job_file.h`, `print_action_dispatch.h`, and `print_state_rules.h` depending on surface. | Continue-anyway, preheat status, and preheat abort require representative hardware proof through LCD, web/API, and Cura. |
| pending-job metadata files | Shared library/API boundary. | Shared `pending_job_file.*`, `pending_job_dispatch.*`, `print_job_file.*`, and print-service `pending_job_registration.*`. | `/tmp` metadata remains a migration boundary between upload handlers, LCD prompts, and native `JOB` dispatch. | Replace file-backed handoff with a native service or shared local API once all clients can query one authoritative pending-job state. | Shared helpers are used by LCD, web/API, Cura routes, and native print-service registration. | Restart recovery and stale pending-job clearing must be proven on hardware for USB, web, and Cura paths. |
| direct macro calls | Native service-owned. | Native `deneb-printsvc` macro registry and Deneb-owned `/etc/deneb/marlindriver/gcode` defaults. | Clients may still send `MACRO` requests over the stock-shaped command frame. | Preserve the command frame only; macro resolution and safety must stay service-owned. Remove any client-side fallback to stock macro paths. | Native macro registry, packaged `deneb-printsvc-macros/`, package audit, and release archive audit reject Python driver artifacts. | Hardware proof is still required for every physical macro class before broad parity is claimed. |
| direct raw G-code calls | Native service-owned with shared planning helpers. | Shared `command_format.*`, `gcode_command.*`, `manual_motion.*`, native `gcode_rewrite.*`, `heater_wait.*`, and service command dispatch. | REST/LCD clients may issue stock-shaped `GCODE` requests for compatibility, but command planning must live in helpers or native service modules. | Move raw string assembly into shared helpers or service policies. Any remaining direct string send must be a reviewed adapter call. | API printer motion and temperature commands use shared G-code helpers; native service owns raw multi-line sequencing, heater waits, and rewrite policy. | Physical heat/motion proof remains open for `G280`, `M109`, `M190`, and raw multi-line wait behavior. |
| duplicated status classification | Shared library/API boundary. | Shared `print_state_rules.*`, `status_payload.*`, `status_state.*`, `printer_status_response.*`, and service status serialization. | Each client still maps stock-shaped status JSON into its own UI/API representation. | Remove client-local lifecycle heuristics once service status fields are sufficient for active, preparing, stop-allowed, paused, aborting, and idle state. | LCD and web ZMQ clients include `status_state.h`, `status_payload.*` feeds service status, REST status output uses `printer_status_response.h`, and `status_state.*` owns abort-display context, lifecycle transition classification, completion labels, and preheat event classification with host coverage. | LCD/Web UI proof is still required for active/preheat/abort/completion transitions on hardware. |
| diagnostics and error mapping | Native service-owned with shared export helpers. | Native `runtime_diagnostics.*`, `diagnostics_log.*`, `error_map.*`, and shared `diagnostics_export.*`. | Diagnostics remain exposed through touch UI, web/API, smoke scripts, and logs. | Keep shared export surfaces. Remove any duplicated error text mapping when service-owned error codes cover the same condition. | Native source is split into diagnostics, error map, runtime diagnostics, and shared diagnostics export modules. | Live error-path comparison against stock remains open for representative recoverable and fatal printer faults. |
| native `deneb-printsvc` callers | Native service-owned. | Modular native service files own serial transport, packet framing, flow control, command dispatch, job streaming, heater waits, pause/resume, abort/finalize, status parsing, and diagnostics. | The public stock-shaped ZMQ IPC frame remains while the backend implementation is native. | Promotion beyond experimental requires every client path to use native service without stock fallback and current hardware/resource evidence. | `printsvc/src` is split by responsibility and host tests plus package/native audits gate the no-Python route. | Full live-device smoke matrix and stock/native resource comparison remain open. |

The companion `deneb-printsvc-integration-audit` script checks this document and
the source wiring so new client patches have to declare an owner, a migration
boundary, and a removal condition instead of growing another private copy of
print-state logic. It also rejects raw motion/heater G-code string literals in
UI and web/API `.c` client adapters so those commands stay in shared/native
policy owners rather than returning as per-client patches. Direct references to
the pending-job file path are likewise rejected in UI and web/API adapters so
the file-backed migration boundary stays confined to shared/native pending-job
owners.
